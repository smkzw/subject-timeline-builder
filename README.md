# Subject Timeline Builder 使用说明

## 这个 Skill 用来做什么

`subject-timeline-builder` 用于把临床试验中的研究方案、EDC data listing、核查 Finding listing、禁用/限制用药解释、AE/lab/PK/PD listing 等资料，整理成一个中文交互式“受试者时间线”HTML，并在需要时输出风险汇总 Excel。

生成的页面适合临床项目经理、医学经理、PV、稽查/核查准备团队使用，用于快速查看每位受试者在研究期间的访视、AE、病史、既往/合并用药/治疗、Finding、PK/PD 信息，以及潜在禁用药/治疗、入排违背和 AE 判定风险。

## 什么时候会触发

当用户输入中包含以下类似内容时，AI Agent 应使用该 Skill：

- 受试者时间线
- subject timeline
- patient profile timeline
- patient profile HTML
- 按受试者整理访视、AE、合并用药、Finding
- 核查禁用药漏报、入排违背、AE 判定不合理风险
- 生成临床试验受试者 profile 页面

示例：

> 请基于上传的方案、data listing、finding listing、禁用药解释和 PK/PD listing，生成受试者时间线和风险汇总。

> 帮我把这个中心的受试者做成 patient profile timeline。

## 需要上传哪些文件

通常建议上传以下文件：

1. 研究方案
   - PDF、Word、TXT 均可。
   - 用于识别研究周期、访视点、合并用药定义、既往用药定义、现病史/既往史定义、禁用/限制用药规则、入排标准、AE 特别关注事项和 PK/PD 采样规则。

2. Data listing
   - 通常为 EDC 导出的 Excel。
   - 需要包含受试者、访视日期、AE、既往/现病史、既往及合并用药等 sheet。

3. Finding listing
   - 项目自查或核查发现问题清单。
   - 可以是 Excel、CSV 或其他表格文件。
   - 如果没有，也可以不展示 Finding 模块。

4. 禁用/限制用药解释或规则表
   - 可以是方案附录、医学解释表、Excel、CSV 或其他表格。
   - 用于拆解类别、代表药物、洗脱期、半衰期规则、允许/豁免情况、补救治疗规则和是否需要登记 PD/Finding。

5. Lab listing
   - 当入排或安全风险依赖实验室阈值时需要。
   - AI 不应只因病史名称或“实验室异常”字样就判定入排风险，需结合方案阈值和实际筛选/随机化实验室结果。

6. AE/IB/说明书/同靶点风险资料
   - 如果用户要求 AE 判定风险，需要提供或明确指定 AE 风险来源。
   - AI 应区分 AESI、SUSAR、SAR、SAE、≥3 级 AE，以及 IB/说明书/同靶点提示风险。

7. PK/PD listing
   - 可以在 data listing 中，也可以是单独文件。
   - 如果只有“是否采集 PK/PD”的记录，而没有具体浓度/结果，AI 会先请用户确认是仅展示采集状态、另传具体结果 listing，还是不展示该模块。
   - 如果没有，也可以不展示 PK/PD 模块。

8. 受试者分组表或随机/筛选结果表
   - 用于识别受试者治疗组、是否随机、是否筛选失败。
   - 如果上传文件中没有这些信息，AI 会提示补充文件或指出对应 sheet/列。

## AI 会先做哪些检查

正式生成 HTML 前，AI 会进行 precheck：

- 识别每个文件的用途。
- 读取 Excel 中的 sheet 名和列名。
- 判断哪些 sheet 对应访视、AE、病史、用药、PK、Finding。
- 判断是否有非药物治疗、实验室、禁用药解释、PD/Finding 细项、随机/分组信息。
- 检查是否存在重复列名，例如两个“药物名称”列。
- 检查方案中是否明确写了“合并用药”“既往用药”“现病史”“既往史”的定义。
- 检查方案和解释表中是否能拆出禁用/限制用药类别、洗脱期、半衰期规则、允许/豁免情况和补救治疗规则。
- 检查 PD/Finding 是否能按事件级别匹配到具体受试者、具体用药/治疗、具体日期或日期范围，而不是只按受试者粗略匹配。
- 检查入排风险是否有可计算的 lab 阈值；有阈值时需结合实际数值。
- 检查是否缺少关键字段，例如受试者编号、访视日期、AE 名称、用药开始/结束日期等。
- 检查是否能识别随机/筛选结果和治疗分组。
- 检查是否存在 USV/计划外访视，以及是否需要纳入个例时间线。
- 检查 PK/PD 文件中是具体检测结果，还是仅记录是否采集。不会把“是否采集”误当作浓度/结果或默认显示 LLOQ。

## 什么时候需要用户确认

如果 AI 无法从文件中明确判断，会向用户确认。常见情况包括：

1. 方案没有明确合并用药定义

   AI 会询问是否按以下默认规则：

   > 使用日期范围与“首次试验药物用药至治疗期末次访视”重叠的非研究药物，作为合并用药。

2. 方案没有明确现病史定义

   AI 会询问是否按以下默认规则：

   > 疾病日期范围与“首次试验药物用药至整个研究末次访视”重叠的病史，作为现病史。

3. Data listing 中存在多个相似列

   例如 CM sheet 中存在两个“药物名称”列。AI 会提示并确认是否使用靠前的原始录入药名，而不是编码后的标准药名。

4. 缺少某个模块的数据

   例如未提供 PK listing。AI 会询问是否不展示 PK 模块。

   如果 PK/PD sheet 仅记录各访视是否采集样本，AI 会询问是否仅展示采集状态，或要求补充具体结果 listing。

5. 需要确定纳入人群

   AI 会询问：

   > 生成 HTML 时纳入所有受试者，还是仅纳入已随机受试者？

6. 需要确定是否纳入 USV

   AI 会询问：

   > 是否纳入计划外访视（USV）数据？

   如选择纳入，USV 会按该受试者 USV 的实际日期放入个例时间线。

7. 用药原因无法追溯到具体 AE 或病史

   如果既往/合并用药的原因写为“不良事件”或“其他既往及现病史”，但 listing 中无法找到具体 AE/MH 或备注，AI 会提示用户确认。

如果用户选择不展示某个模块，最终 HTML 中不会出现该模块。

## 生成的 HTML 包含什么

最终页面为中文交互式 HTML，通常包含：

- 顶部筛选器
  - 按中心筛选
  - 按受试者筛选
  - 关键词搜索

- 每位受试者一个 profile card
  - 受试者编号
  - 治疗分组、随机状态或筛选状态，按当前研究原始分组/状态展示
  - 所属中心
  - 研究窗口或治疗窗口
  - 用户要求的受试者级风险提示卡片
  - 可视化时间线

- 时间线内容
  - 访视日期和访视点
  - AE / 安全性事件
  - 现病史 / 既往史
  - 合并用药
  - 既往用药
  - 合并非药物治疗

- 表格式/列表式信息
  - AE 名称、起止日期、严重程度、与研究用药关系、转归、SAE 信息
  - 病史名称、起止日期、分类
  - 药物名称、起止日期、频率、途径、治疗原因；如果原因关联 AE 或病史，会显示具体 AE/MH 信息
  - 非药物治疗名称、起止日期、治疗原因
  - PK/PD 访视、采集状态或具体结果；仅在有真实结果列时显示浓度、LLOQ 或定量解释
  - Finding/PD 分类、涉及访视、风险程度、原始描述、建议回复口径

- 风险汇总 Excel（如用户要求）
  - 用药/治疗风险：禁用或限制用药、洗脱不足、补救或豁免状态、PD/Finding 登记状态
  - 入排风险：病史、实验室阈值、既往治疗或其他可由 listing 计算的入排问题
  - AE 判定风险：AESI/SUSAR/SAR/SAE/≥3 级 AE、IB/说明书/同靶点风险提示、可能不合理的无关性判定
  - 每行保留受试者、编号、源 sheet/源行、匹配规则、证据文本和判定理由

## 页面样式

HTML 采用适合核查准备和医学审阅的样式：

- 中文界面。
- 浅色背景，信息密度适中。
- 每位受试者独立卡片展示。
- 时间线用颜色区分 AE、病史、合并用药。
- 风险色块和圆角标签用于提示不同风险状态，但不改变核心字号、行距和信息密度。
- 右上角筛选器可快速定位中心和受试者。
- 鼠标悬停在时间线条目上可查看详细信息。
  - AE 色块显示事件名称、日期、严重程度、相关性、转归等。
  - 病史色块显示疾病名称、日期、是否持续等。
  - 用药色块显示药物名称、日期、剂量、频率、途径、用药原因等。
- 可直接在浏览器中打开，无需安装额外软件。

## 使用建议

首次使用时，建议按以下顺序给 AI：

1. 上传研究方案、data listing、finding listing（如有）、PK listing（如有）、受试者分组表（如有，或确认 data listing 中包含随机/筛选结果）；
2. 明确说明是否要包含所有中心、所有受试者。
3. 确认纳入所有受试者还是仅纳入已随机受试者。
4. 确认是否纳入 USV/计划外访视。
5. 等待 AI 进行 precheck。
6. 对 AI 提出的缺失字段或定义问题进行确认。
7. 让 AI 生成最终 HTML 和风险汇总 Excel。

## 脚本能力边界

仓库内 `scripts/build_timeline_html.py` 是通用基础时间线生成器，适合在列映射清晰时快速生成访视、AE、MH、CM、Finding、PK 等基础模块。复杂项目中的禁用药/治疗、入排、AE 判定风险通常需要 AI 基于当前方案、解释表和 listing 做项目本地扩展或包装脚本，并额外输出风险汇总 Excel 和 QC 记录。不要把基础脚本未实现的风险逻辑当作已经自动覆盖。

## 注意事项

- Skill 不会固定使用任何历史项目规则。
- 每次都会基于新上传的方案和 listing 重新判断。
- 如果方案和 listing 之间存在冲突，AI 会优先提示用户确认。
- 如果日期为不完整日期，例如 `2024-05-uk`，AI 会在分类时按保守规则处理，并保留原始日期文本用于展示。
- 如果 listing 中包含编码药名和原始录入药名，默认应优先展示原始录入药名，除非用户明确要求展示编码名。
- PD/Finding 登记状态必须按事件级别匹配，不可因为同一受试者存在相似事件就把所有同类事件标记为已登记。
- 对于禁用/限制用药，补救治疗或方案允许的豁免情况仍可作为风险提示保留，但不应直接归为潜在漏报。
- 对于实验室相关入排风险，必须结合方案阈值和实际实验室结果；不能仅凭病史名称或异常描述推断。
- 对于 AE 判定风险，通常只有“无关/可能无关”且匹配到已提供的 IB、说明书、方案或同靶点风险时，才进入潜在判定不合理提示；已判定相关的事件只做风险来源提示。
